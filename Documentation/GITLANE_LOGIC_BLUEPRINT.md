# GitLane Logic Blueprint

This document translates the SpectraGuard web flow into an offline-first React Native experience anchored by `isomorphic-git`, `expo-file-system`, and `react-native-mmkv`. Each section ties back to the source files you asked about so you can align new UI components with concrete expectations for identity, data, state, transactions, and offline sharing.

---

## Identity Mapping

- **Current Flow:** `AuthContext.jsx` polls `/api/auth/status` (with `credentials: include`) to learn if the session is authenticated and then pushes the GitHub `user` object into React state; `SignInModal.jsx` kicks off the OAuth dance via `window.location.href` and listens for `message` events that carry `oauth-success` before refreshing the auth state and navigating to `/dashboard`.
- **Mobile translation:**
  - Replace repeated backend status checks with a one-time OAuth exchange that ultimately stores the GitHub identity in MMKV. After successful OAuth (you can still redirect to a WebView or SFSafariViewController inside Expo), persist these fields in MMKV keys such as `gitlane:userLogin`, `gitlane:userName`, `gitlane:userEmail`, and `gitlane:authToken` so downstream code never needs cookies.
  - Mirror `signOut` by wiping the MMKV namespace (clear `user` keys and any `transactionLog` entries) and navigating to the landing stack.
  - Expose a custom context hook similar to `useAuth` but pointing at MMKV values and exposing `refreshFromMMKV` instead of `refreshAuthStatus`. Keep `isLoading`, `isAuthenticated`, and `user` states for compatibility with the new UI; derive them from MMKV on mount and subscribe to MMKV events if your wrapper supports it.

## Data Structure Translation

| GitHub API payload | Example fields | Isomorphic-git equivalent | Notes for GitLane |
| --- | --- | --- | --- |
| **Repositories list** (from `/api/repos`, consumed by `dashboard.jsx` and `RepoList.jsx`) | `{ id, name, full_name, private, owner: { login, avatar_url }, description, language, stargazers_count, forks_count, updated_at }` | `git.listRemotes({ fs, dir })` + `fs.readdir(dir)` per remote to discover repo contents. Actual repo metadata derives from the `.git/config` remotes (`name`, `fetch`, `url`) and from `fs.stat`/`git.log` to compute last updated timestamp. | Treat the `owner.login` value as your locally stored `userLogin`; expose `description`, `language`, and visibility flags by inspecting `repo.git/config` or storing them in MMKV metadata when the repo was cloned.
| **Repository contents** (`/contents`, consumed by `FileList.jsx`) | `[ { name, path, sha, size, type ('file'|'dir'), download_url } ]` | `git.listFiles({ fs, dir })` returns flattened relative paths; `git.readBlob` gives `oid`, `blob` buffer, and `type` (`blob` or `tree`). Use `fs.stat` for size and `type === 'tree'` for directories. | Build a small adaptor that turns `git.listFiles` into an array of objects with `name`, `path`, `type`, `oid`, and `size` so your performant UI components (cards, tables) can reuse the same props.
| **Commits/history** (`history.jsx` fetches `/api/history/repo/:id`) | `[ { id, createdAt, securityScore, results, repository: { full_name } } ]` | `git.log({ fs, dir, depth })` returns `commits` with `{ oid, commit: { message, author: { name, email, timestamp }, committer, tree } }`. | Store local history entries in MMKV with the `oid`, `author.name`, and timestamp. Combine them with the security metadata generated by your in-app scans to present `securityScore` and `status` badges similarly.
| **Remotes** | (Not directly fetched in the frontend but implicit in repo metadata) | `git.listRemotes({ fs, dir })` returns `[ { remote: 'origin', url: 'https://github.com/owner/repo.git' } ]`. | Persist the remote list inside MMKV after clone to rehydrate dashboards without rereading `.git/config` on every render.

```json
  }
}
```

## State Management (Loading + Error)

- Replace every `fetch(...)` call (e.g., `/api/repos` in `RepoList`, `/api/repos/:owner/:repo/contents` in `FileList`, `/api/history/...` and `/api/repo/...` in `History`, and profile/repos calls in `dashboard.jsx`) with local `isomorphic-git` steps:
  1. Ensure `fs` and `dir` point to your Expo-managed directory (e.g., `FileSystem.documentDirectory + repoName`).
  2. Use `git.listRemotes`, `git.log({ depth: 1 })`, `git.listFiles`, or `git.statusMatrix` depending on the screen’s needs.
  3. Wrap each async operation in a shared `useAsyncResource` hook that exposes `{ data, loading, error, refresh }` just like the current components do.

- Define consistent states:
  - `loading`: `true` while any `isomorphic-git` promise is unresolved.
  - `error`: populated with the thrown message when git operations fail (e.g., directory missing, corrupted `.git`). Use this for the red error banners already in the UI.
  - `data`: e.g., `repos`, `files`, `historyRecords`.
  - `selectedRepo`: keep this UI contract so the RepoList/FileList combo still works; `FileList` can observe `selectedRepo.dir` and refresh when it changes.

- Use toast/focus handling analogously to the existing `History` download button: show ephemeral messages when a git command fails (e.g., `git.log` rejects due to missing repo) or when operations succeed.

## Offline "Transaction" Logic

- The backend controller (`controllers/scan.controller.js`) defines a four-stage scan pipeline: clone the repo, run four layered scans (`scanForSecrets`, `scanDependencies`, `scanWithAI`, `scanCommitHistory`), calculate security score, persist via Prisma, and return a `scanId` plus full payload. Translate that into a **mobile transaction log stored in MMKV**.

- Proposed mobile transaction log schema (store JSON per repo):
  ```json
  {
    "repoFullName": "owner/repo",
    "transactions": [
      {
        "id": "<git-oid-or-timestamp>",
        "type": "scan|commit|sync",
        "status": "queued|cloning|scanning|committing|completed|failed",
        "layerStatus": {
          "layer1": "pending|done|error",
          "layer2": "pending|done|error",
          "layer3": "pending|done|error",
          "layer4": "pending|done|error"
        },
        "error": "if any",
        "updatedAt": 171938…
      }
    ]
  }
  ```
  Use MMKV keys like `gitlane:transactions:<repoFullName>` and a master index `gitlane:transactionIndex` for quick lookups.

- Workflow:
  1. When the mobile UI triggers a scan (equivalent to the POST body in `startScan`), push a new transaction entry with `status: 'queued'` and `layerStatus` all `pending`.
  2. As each `isomorphic-git` step runs (clone/pull, run binary scanners, commit, etc.), update MMKV with `status: 'cloning'` -> `status: 'scanning'` -> `status: 'completed'`. Also persist each layer’s completion so you can resume mid-scan.
  3. On app cold start, read all `queued`/`cloning`/`scanning` entries, verify whether the corresponding repo directory still exists, and continue from the last `layerStatus` (e.g., rerun layer 3 if it failed before completion). This mirrors the backend cleanup/continue logic before it deletes `tempDir`.
  4. Provide a `Transaction Log` screen (not yet implemented) that surfaces each transaction’s current status (use the same color scale as `getScoreColor` from `reportDownloader.js`) so users understand whether a scan is in-flight, succeeded, or needs manual retry.

- Recovery: When a transaction is flagged `failed`, keep the temp directory and git state intact so a subsequent attempt can `git.reset` or `git.checkout('main')` before picking up where it left off.

## P2P Logic (Exporting as `.bundle`)

- `reportDownloader.js` currently turns the scan into a PDF by iterating through `results.layer1…4` and rendering tables. For GitLane, apply the same layer-based reporting idea to bundling a repo for offline sharing.
  1. Use `isomorphic-git`’s `pack`/`writeObject` utilities (or spawn `git` if you bundle via `git bundle` in a native module) to create a `.bundle` that contains the repository history.
  2. Mirror the helper layering: gather metadata (layer1 → repo/remote info, layer2 → dependency tree), add a summary page (maybe a JSON file) describing the transaction log (from MMKV) and the current security score.
  3. Write the bundle to Expo’s file system (e.g., `${FileSystem.documentDirectory}bundles/${repoFullName.replace('/', '__')}.bundle`).
  4. Once written, expose it via `expo-sharing` or `expo-file-system`’s `getContentUriAsync` so the user can AirDrop/Share it directly; optionally zip it first for compatibility with `jszip`.

- Provide a lightweight manifest (JSON) next to the bundle that replicates the structure generated inside `downloadPdfReport`: include `score`, `deductions`, `layerStatus`, and the MMKV `Transaction Log` entry used to create the bundle. This makes it easy for another GitLane installation to ingest the bundle, verify which layers were finished, and resume if needed.
